# Unified Pass-Through Trace

This document defines the decoded trace consumed by
`tools/validate_passthrough_trace.py`.

The trace is not the raw UDP stream and not the raw HDMI image stream. It is the
common evidence format between a future hardware runner and the reusable
validator:

```text
PC sender metadata + decoded HDMI capture metadata -> trace JSON -> validator
```

## Schema

```json
{
  "schema": "unified-passthrough-trace-v1",
  "requirements": {
    "min_match_rate": 0.95,
    "max_drop_rate": 0.05,
    "max_latency_ms": 250.0,
    "max_order_violations": 0,
    "max_content_mismatches": 0,
    "max_black_frames": 0,
    "min_luma": 8.0,
    "require_image_paths": false
  },
  "sent": [
    {
      "frame_id": 0,
      "sent_ms": 0.0,
      "content_id": "frame-000000"
    }
  ],
  "captured": [
    {
      "capture_index": 0,
      "captured_ms": 120.0,
      "decoded_frame_id": 0,
      "content_id": "frame-000000",
      "mean_luma": 120.0,
      "image_path": "images/capture-000000.ppm",
      "image_sha256": "optional sha256 of image_path"
    }
  ]
}
```

## Field Meaning

- `frame_id`: the frame sequence number generated by the sender.
- `sent_ms`: sender timestamp in milliseconds from the trace start.
- `content_id`: an identifier for the intended visual payload of that frame.
- `captured_ms`: HDMI capture timestamp in milliseconds from the same trace
  origin, after clock alignment by the hardware runner.
- `decoded_frame_id`: frame id decoded from the captured HDMI image. A null
  value means no frame id was recovered.
- `mean_luma`: measured brightness of the decoded capture frame. It prevents
  black/no-frame output from being counted as a valid correspondence.
- `image_path` and `image_sha256`: optional evidence fixture. If
  `require_image_paths` is true, the validator requires the image file to exist
  and match the hash when provided.

## Hardware Image Evidence

For hardware pass-through cycles, `decoded_frame_id` must come from the saved
HDMI image or an equivalent offline re-decode of the captured image evidence.
It must not be inferred only from sender order or from a repeating color
sequence.

The 2026-07-01 hardware runner uses a deterministic generated RGB888 source
with a small synchronized black/white frame marker:

```text
sync cells: black, white
data cells: 12 bits, least significant bit first
cell geometry: recorded in trace.alignment.marker
```

The trace builder accepts a captured frame only when the sync cells decode
correctly. The captured `content_id` is derived from the decoded frame ID and
decoded color, so a wrong color becomes a validator `content_mismatch` instead
of being hidden by the runner.

`max_latency_ms` is a per-trace requirement. For HDMI-UVC/MJPEG evidence paths
it measures the return-capture chain, not board-internal processing latency.
The requirement value used for a hardware cycle must be recorded in that
cycle's report along with the measured max latency.

## Pass Semantics

The validator checks temporal correspondence, not visual plausibility:

```text
matched_frames / sent_frames >= min_match_rate
drop_rate <= max_drop_rate
max matched latency <= max_latency_ms
order_violations <= max_order_violations
content_mismatches <= max_content_mismatches
black_frames <= max_black_frames
```

`drop_rate` is computed from integer frame counts:

```text
(sent_frames - matched_frames) / sent_frames
```

It is not computed as `1.0 - match_rate`, so exact boundary cases such as
19/20 matched frames remain stable at `0.05`.

Frame order is checked only after a captured frame has been matched to a sent
frame and is not a duplicate. Unmatched captures still fail validation, but they
must not create spurious `frame_order_violation` failures for later legitimate
frames.

This deliberately replaces weaker ad-hoc checks such as non-black luma, unique
JPEG hashes, or color-set equality. Future hardware cycles should use this trace
format when claiming a faithful live pass-through loop.

## Calibration

The validator-introduction cycle calibrates the script with synthetic traces:

```powershell
rtk powershell.exe -NoProfile -Command "python .\tools\validate_passthrough_trace.py --calibration --out-dir build\unified-passthrough-validator-calibration"
```

The calibration must include:

```text
known_good_pass
known_bad_black_fail
known_bad_wrong_order_fail
known_bad_missing_frame_fail
known_bad_wrong_content_fail
known_bad_latency_fail
```

Only after that calibration is committed may a later hardware cycle use the
validator as its frozen pass gate.
